# Cockpit-NG Backend — Code Analysis (Fable)

**Date:** 2026-06-09
**Scope:** `/backend` Python code (~713 files, ~153k LOC including tests)
**Reviewer:** Backend analysis pass covering security, Python best practices, and architecture/refactoring needs.

---

## Executive Summary

The backend is in **good overall shape**. The team has clearly invested in security hardening (startup secret pre-flight checks, Fernet credential encryption, sanitized 5xx errors, rate-limited login, regression guard scripts) and a disciplined layered architecture. Most of the "INCORRECT Practices" listed in `CLAUDE.md` are actually *enforced* — f-string logging is essentially eliminated (1 hit, in a checker docstring), no `text()` SQL leaks into routers/services, no hardcoded secrets, no `eval`/`exec`/`pickle`.

The findings below are mostly **MEDIUM/LOW** refinements, with a small number of **HIGH** items worth prioritizing. There is no critical, exploitable-by-default vulnerability in the code reviewed.

| Severity | Count | Theme |
|----------|-------|-------|
| HIGH     | 3     | `shell=True` subprocess, token-in-URL git auth, deterministic KDF salt |
| MEDIUM   | 6     | Path-prefix check, API key hashing, blocking I/O in async, large files, `utcnow()`, coverage gate |
| LOW      | 5     | jose+pyjwt duplication, unpinned deps, Jinja sandbox, debug router exposure, broad excepts |

## 1. Security Risks

### 1.1 HIGH — `shell=True` with f-string command construction (`fping`)

**Files:** `tasks/ping_network_task.py:104-109`, `services/network/scanning/network_scan.py:311-314`

```python
full_cmd = f"{' '.join(cmd)} < {temp_file_path}"
result = subprocess.run(full_cmd, shell=True, capture_output=True, timeout=60, text=True)
```

`shell=True` is used solely to get input redirection (`< tempfile`). The `temp_file_path` is generated by `tempfile.NamedTemporaryFile` (safe today), and the IP list is written to the file rather than the command line, so this is **not currently injectable** by user input. However:

- It is a latent injection vector — any future refactor that interpolates a user-controlled value into `cmd` or the path becomes an instant RCE.
- It violates the principle of least surprise and will be flagged by every SAST tool.

**Fix:** Drop the shell entirely. `fping` reads targets from stdin with `-f -` or accepts a file with `-f <file>`. Use the argument-list form and pass the file via `stdin` or `-f`:

```python
cmd = ["fping", "-c", str(count), "-t", str(timeout), "-r", str(retry),
       "-i", str(interval), "-f", temp_file_path]
result = subprocess.run(cmd, capture_output=True, timeout=60, text=True)  # no shell
```

This removes `shell=True` from both call sites.

### 1.2 HIGH — Git token written into the remote URL

**File:** `services/git/service.py:296-322` (and the parallel push path ~376-408)

For token auth the code temporarily rewrites the remote URL to embed the credential (`origin.set_url(auth_url)` where `auth_url` contains the token), then restores it in a `finally`. Risks:

- The token (in `https://user:token@host/...` form) can leak into **GitPython error messages**, **git's own stderr**, and any log line that prints the URL.
- The restore in `finally` swallows exceptions silently (`except Exception: pass`), so a crash mid-operation can leave the token persisted in `.git/config`.

**Fix:** Prefer credential injection that never touches the URL or on-disk config:
- Use `GIT_ASKPASS` / a short-lived credential helper, or
- Pass the token via the `http.extraHeader` config for the single command (`-c http.extraHeader="Authorization: Basic ..."`), or
- At minimum, ensure the `auth_url` is never logged and that the restore failure is logged + alerted rather than `pass`ed.

Also verify `PullResult(message=f"Pull failed: {str(e)}")` (line ~327) cannot echo an auth URL back to the client.

### 1.3 HIGH (design) — Deterministic, hardcoded KDF salt for credential encryption

**File:** `core/crypto.py:22-34`

```python
_KDF_SALT = b"cockpit-credential-encryption-v2"
_KDF_ITERATIONS = 100_000
```

The encryption design is otherwise sound (Fernet = AES-128-CBC + HMAC, PBKDF2-HMAC-SHA256). Two concerns:

1. **Fixed salt** means the KDF output depends only on `SECRET_KEY`. A fixed salt defeats the salt's purpose (precomputation resistance across deployments) — every Cockpit install with the same `SECRET_KEY` derives the same key. Since `SECRET_KEY` is also the JWT signing key, **one secret protects both auth tokens and credential-at-rest encryption**; compromise of `SECRET_KEY` exposes all stored device passwords/SSH keys.
2. **100k PBKDF2 iterations** is below the OWASP 2023 recommendation (600k for PBKDF2-HMAC-SHA256). Less critical here since the input is a high-entropy secret rather than a user password, but worth raising.

**Recommendations (defense in depth):**
- Use a **separate** key for credential encryption (e.g., `CREDENTIAL_ENCRYPTION_KEY`) so JWT-key rotation and credential-key rotation are independent.
- Keep the salt fixed *only* because the design requires deterministic keys — document this explicitly (the comment already warns), but consider per-record salts stored alongside ciphertext for new credentials.
- Bump iterations toward 600k, paired with a re-encryption migration.

The code already centralizes this correctly ("never define local EncryptionService classes") — that discipline is good and should be preserved.

### 1.4 MEDIUM — Path-containment check uses string prefix, not path-boundary

**File:** `services/git/file_service.py:389-393`

```python
if not file_path_resolved.startswith(repo_path_resolved):
    raise HTTPException(status_code=403, detail="...outside repository")
```

`realpath` + `startswith` is the right *idea*, but a bare prefix match is vulnerable to sibling-directory bypass: `/repos/myrepo-secret` starts with `/repos/myrepo`. Exploitability depends on the repo layout, but it's a known footgun.

**Fix:** Compare with a trailing separator or use `Path`:
```python
repo_root = Path(repo_path).resolve()
target = Path(file_path).resolve()
if repo_root != target and repo_root not in target.parents:
    raise HTTPException(403, ...)
```
Apply consistently to all read/write/list/delete paths in this file (multiple `os.path.join(repo_path, path)` sites: lines 387, 457, 533, 640, 942, 1234).

### 1.5 MEDIUM — API keys stored and looked up in plaintext

**Files:** `core/models/users.py:47` (`api_key = Column(String(255))`), `core/auth.py` (`profile_repo.get_by_api_key(x_api_key)`)

API keys appear to be stored verbatim and matched by direct equality lookup. If the `user_profiles` table leaks, all API keys are immediately usable. Equality lookup on a plaintext column is also (theoretically) timing-observable.

**Fix:** Store a hash of the API key (SHA-256 is acceptable for high-entropy random keys; the key itself need not be slow-hashed). Look up by hash. Show the raw key to the user only once at creation time. This mirrors how GitHub/Stripe handle API tokens.

### 1.6 MEDIUM — Blocking `requests` calls inside `async def` handlers

**Files:** `services/checkmk/client.py`, `services/checkmk/client/_connection_service.py`, `services/nautobot/onboarding/onboarding_service.py`, `routers/nautobot/devices.py`

Mixed picture: the CheckMK inventory path **correctly** wraps the blocking call in `asyncio.to_thread` (`_connection_service.py:158`). But several `requests.get/post` calls live in modules that also define `async def`. Any synchronous `requests` call executed directly on the event loop **blocks the entire FastAPI worker** for the duration of the network round-trip, serializing all concurrent requests.

**Fix:** Audit each `requests.*` call reachable from an `async def`. Either (a) move to `httpx.AsyncClient` (already a dependency), or (b) wrap in `asyncio.to_thread` as `_connection_service.py` does. Standardize on one HTTP client for external calls.

---

### 1.7 LOW — Two JWT libraries in use

`core/auth.py` uses **PyJWT** (`import jwt`); `services/auth/oidc.py` uses **python-jose** (`from jose import JWTError, jwt`). Both are in `requirements.txt`. Maintaining two JWT stacks doubles the CVE surface (python-jose has had several advisories) and invites subtle algorithm-handling differences. Consolidate on one — PyJWT is the more actively maintained choice.

### 1.8 LOW — Debug router exposes filesystem read/write/delete

**File:** `routers/git/debug.py` — `POST /{repo_id}/debug/read|write|delete|push`

These are guarded by `require_permission("git.repositories", "write")`, which is correct, but a "debug" router that performs arbitrary repo file writes/deletes in production is a large attack surface. Recommend gating the whole router behind a settings flag (`ENABLE_DEBUG_ENDPOINTS`) that defaults to off in production.

### 1.9 LOW — CORS / proxy trust

CORS is not configured in `main.py` (no `CORSMiddleware`), consistent with the documented "frontend always calls via `/api/proxy/*`" model — the backend is meant to be unexposed. Worth a one-line confirmation in deployment docs that the backend port is never published directly, since there is no in-app origin enforcement to fall back on.

## 2. Python Best Practices

**Strong adherence overall.** Confirmed positives:

- ✅ **Logging:** No f-string logging in application code (`%s` lazy-formatting used throughout). The single grep hit is a docstring in `scripts/check_fstring_logging.py`.
- ✅ **No `eval`/`exec`/`os.system`/`pickle`/`yaml.load`** in application code.
- ✅ **No hardcoded secrets** (the two matches are a test client's `password="automation"` against a mock, and an enum value `TOKEN = "token"`).
- ✅ **Sanitized 5xx errors** via `core.safe_http_errors.raise_internal_server_error` — no `str(e)` leaking in server-error details (the regression guard `check_http_500_leaks.py` enforces this).
- ✅ **Startup pre-flight** rejects the default `SECRET_KEY` and warns on default admin password (`main.py:352`).
- ✅ **Rate limiting** on `/auth/login` (`@limiter.limit("10/minute")`).
- ✅ **TLS verification on by default** (`verify_ssl: bool = True` in CheckMK client and settings models).
- ✅ **`from __future__ import annotations`** + `TYPE_CHECKING` pattern used in core modules.

### Issues to address

**2.1 MEDIUM — `datetime.utcnow()` is deprecated (Python 3.12+) and returns naive datetimes.**
~25 occurrences across application code (`set_admin_password.py`, `health.py`, `routers/auth/oidc.py`, `routers/inventory/crud.py`, `repositories/cockpit_agent/...`, `repositories/jobs/job_run_repository.py`, `services/settings/credentials_service.py`, `services/network/...`, etc.). `core/auth.py` already does the right thing (`datetime.now(timezone.utc)`). Mixing naive and aware datetimes risks comparison `TypeError`s and incorrect ordering.
**Fix:** Replace all `datetime.utcnow()` with `datetime.now(timezone.utc)`; add a ruff rule (`DTZ`) to block reintroduction.

**2.2 LOW — Broad `except Exception` / silent `except: pass`.**
Several spots swallow exceptions (e.g. the git URL-restore `finally: except Exception: pass` in `service.py:320`). Where a failure has security or data-integrity consequences (credential cleanup, URL restore), log at `warning`/`error` rather than passing silently.

**2.3 LOW — Unpinned dependencies.**
`requirements.txt` has **41 unpinned** packages (only floor constraints like `cryptography>=42.0.0`, most with no version at all). This makes builds non-reproducible and lets a transitive update break or backdoor a deploy.
**Fix:** Pin exact versions (`==`) and use a lockfile (`pip-tools`/`uv`/`poetry`). Add `pip-audit` or Dependabot to CI.

**2.4 LOW — Jinja2 renders without sandboxing.**
`services/templates/template_service.py:324` uses `Environment(loader=BaseLoader())` (also `Template(...)` in `render_orchestrator.py`, `netmiko.py`). Templates are authored by authenticated users with template permissions, so SSTI is a privileged-user risk rather than anonymous RCE — but config templates rendered server-side warrant `jinja2.sandbox.SandboxedEnvironment` to contain `{{ ''.__class__.__mro__ ... }}`-style escapes.

## 3. Architecture & Refactoring (Large Files / God Objects)

The layered architecture (Model → Repository → Service → Router) and the Nautobot resolver/manager/facade split are well-conceived and largely followed. Regression guards (`check_router_repositories.py`, `check_text_sql.py`, `check_asyncio_run.py`) actively defend the boundaries. Refactoring needs are about **file size and router thickness**, not structural rot.

### 3.1 Files exceeding the 800-line guideline (non-test)

| File | Lines | Assessment |
|------|-------|------------|
| `services/network/tools/baseline.py` | 1510 | **Refactor.** Top of the file is module-level pure functions + `BaselineImportService` (~22 methods). Split helpers into a `baseline_normalizers.py` and keep the service focused. |
| `services/git/file_service.py` | 1279 | **Refactor.** Handles list/read/write/diff/history across many path operations. Split read vs. write vs. history/diff into separate services; centralize the path-containment guard (see §1.4) into one helper. |

Split into `clusters.py`, `virtual_machines.py`, `virtual_interfaces.py`. Endpoints like `create_virtual_machine` (l.339) also carry orchestration/logging that belongs in the manager/service. |

| `routers/settings/rbac.py` | 883 | **Thin out.** Router this size usually means business logic that belongs in `rbac_service`. |

### 3.2 God objects?

No true God Object found. `DeviceCommonService` is a deliberate **facade** (documented and acceptable). The large services above are *fat but cohesive* — they do one domain's work, just at too coarse a grain. The fix is mechanical extraction, not redesign.

### 3.3 Router thickness

Several routers exceed what "thin routers that delegate to services" implies (`clusters.py` 1000, `rbac.py` 883, `jobs/runs.py` 743). `create_virtual_machine` is illustrative: extensive `logger.debug` of request fields, software-image-file list coercion, and result assembly live in the handler. Push this into `VirtualMachineManager`/a VM service so the router only validates input, calls the service, and shapes the response.

### 3.4 Test coverage

181 test files exist and `pyproject.toml` configures `--cov` for services/models/tasks/utils — but I did **not** find a `--cov-fail-under` gate. The global standard is 80%. **Recommend adding `--cov-fail-under=80`** (or the agreed threshold) to `pyproject.toml` so coverage regressions fail CI rather than being advisory.

## 4. Prioritized Action List

**Do first (HIGH):**
1. Remove `shell=True` from `fping` calls — use `-f <file>` argument form (§1.1).
2. Stop embedding git tokens in remote URLs; never log auth URLs; don't silently `pass` on URL restore (§1.2).
3. Separate the credential-encryption key from `SECRET_KEY` and document/justify the fixed salt; raise PBKDF2 iterations with a re-encryption migration (§1.3).

**Next (MEDIUM):**
4. Harden path-containment to a path-boundary check across `file_service.py` (§1.4).
5. Hash API keys at rest; look up by hash (§1.5).
6. Wrap or replace all blocking `requests` calls reachable from `async def` (§1.6).
7. Replace every `datetime.utcnow()` with `datetime.now(timezone.utc)`; add ruff `DTZ` (§2.1).
8. Add `--cov-fail-under` gate (§3.4).

**When convenient (LOW):**
9. Consolidate on a single JWT library (drop python-jose) (§1.7).
10. Pin dependencies + add `pip-audit`/Dependabot (§2.3).
11. Use `SandboxedEnvironment` for user-authored templates (§2.4).
12. Feature-flag `routers/git/debug.py` off in production (§1.8).
13. Split the 800+ line files in §3.1 (mechanical extraction).

## 5. What's Already Good (keep doing it)

- Centralized crypto with an explicit "no local EncryptionService" rule.
- Sanitized 5xx errors + automated regression guard.
- Lazy `%s` logging enforced project-wide.
- Startup security pre-flight (rejects default secret, warns on default password).
- Login rate limiting.
- TLS verification on by default, with opt-in custom CA support for OIDC.
- Repository/Service/Router layering with guard scripts that prevent `text()` SQL and repo-bypass from regressing.
- The Nautobot resolver/manager/facade pattern — a clean way to wrap an external API without forcing it into the repository mold.
